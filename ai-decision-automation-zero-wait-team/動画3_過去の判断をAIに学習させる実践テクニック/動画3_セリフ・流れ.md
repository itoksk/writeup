# 動画3: 過去の判断をAIに学習させる実践テクニック（10分）

## 導入フレーズ（0:00-1:00）
「動画2でChatGPTにあなたの『判断基準』をインストールしました。でも、まだ足りないものがあります。それは『経験』です。あなたが今まで行ってきた数百、数千の承認判断には、言葉にできない『勘』や『パターン認識』が含まれています。今日は、エクセルに眠る過去の承認データを使って、AIを一気に『10年選手』に成長させる『Few-shot学習』の実践テクニックをお教えします。たった30分の作業で、判断精度が70%から90%以上に跳ね上がります」

## Before/After（1:00-2:30）
**Before: 基本プロンプトのみ**
- 一般的な判断基準で動作
- 会社特有の文化や慣習を知らない
- 過去の失敗パターンを学習していない
- 判断精度70%（10件中3件は要修正）

**After: 過去データで学習済み**
- あなたの会社の実際の判断パターンを記憶
- 成功事例・失敗事例から学習
- 微妙な案件も過去の類似例から判断
- 判断精度90%以上（ほぼ人間と同等）

## Few-shot学習の仕組み（2:30-4:30）
「Few-shotとは『少ない事例で賢くなる』という意味です。実際にやってみましょう」

### 3ステップで実現

**Step 1: エクセルで過去データ整理（画面共有）**
```
過去の承認履歴.xlsx を開く：
A列: 依頼内容
B列: 判断結果（承認/却下/条件付き）
C列: 判断理由
D列: その後の結果（成功/失敗/期待通り）
E列: 学習優先度（高/中/低）
```

**Step 2: 代表事例の選定**
```
エクセルでフィルター実演：
- 承認→成功: 上位5件を選択
- 承認→失敗: 上位3件（失敗から学ぶ）
- 却下→正解: 上位3件
- 条件付き: 上位2件
合計13事例を厳選
```

**Step 3: プロンプトへの組み込み（Make画面）**
```
ChatGPTモジュールのSystem Messageに追記：

## 過去の判断事例（参考にしてください）

【承認・成功事例】
1. SFAツール導入30万円 → 営業効率35%向上、5ヶ月でROI達成
2. オンライン研修15万円 → 全社スキル向上、離職率10%改善
3. MA導入25万円 → リード獲得2.5倍、売上20%増

【承認・失敗事例】
1. SNS広告50万円 → 効果は想定の30%、ターゲティング失敗
2. 高額セミナー参加20万円 → 実務に活かせず、費用対効果なし

【却下・正解事例】
1. 展示会出展100万円 → 他社実績も低く、却下が正解
2. 不明確なコンサル60万円 → 具体性なく、却下で問題回避
```

## 実践デモ（4:30-7:00）
「では、実際の効果を見てみましょう」

### デモシナリオ
1. **事例なしで判断テスト**
   - 「新規Webマーケツール月5万円」をSlackに投稿
   - AIの回答：一般的な判断で曖昧

2. **事例ありで同じ案件**
   - 同じ依頼を再度投稿
   - AIの回答：「過去のMA導入成功事例と類似。ただしSNS広告の失敗も考慮し、小規模テストから開始を推奨」
   
3. **精度の違いを可視化**
   - 判断の具体性が向上
   - 過去事例への言及で説得力UP
   - リスク回避の精度も向上

### エクセルからの効率的な転記方法
```
エクセルの便利技:
1. CONCATENATE関数で事例文を自動生成
   =CONCATENATE("案件: ",A2," → ",D2)
   
2. 優先度でソートして上位を選択

3. テキストエディタに貼り付けて整形
```

## データ活用の実践テクニック（7:00-8:30）

### 1. 事例選定のコツ
```
優先すべき事例:
✅ 金額が近い
✅ 時期が新しい（1年以内）
✅ 結果が明確
✅ 判断理由が詳細

避けるべき事例:
❌ 特殊すぎる案件
❌ 個人的な事情が絡む
❌ 結果が曖昧
```

### 2. 事例の更新サイクル
```
月次メンテナンス（5分作業）:
1. 新しい判断結果をエクセルに追加
2. 古い事例を削除（1年以上前）
3. 効果の高い事例を上位に
4. Makeでプロンプト更新
```

### 3. セキュリティ配慮
```
必須の匿名化:
- 個人名 → 部署名に変更
- 具体的な顧客名 → 業界名に
- 機密金額 → レンジ表記に
```

### 4. Make変数での管理
```
上級テクニック:
- 事例部分を変数化
- 部署別に異なる事例セット
- A/Bテストで最適な事例数を検証
```

## 実装の注意点（8:30-9:00）

### よくある失敗と対策
| 失敗 | 原因 | 対策 |
|------|------|------|
| 事例が多すぎて混乱 | 20件以上入れる | 10-15件に厳選 |
| 古い事例で誤判断 | 3年前のデータ使用 | 1年以内に限定 |
| 特殊事例で一般化 | レアケースを含める | 典型例のみ選択 |

### 最適な運用方法
- 初期：5-7事例から開始
- 2週間後：10事例に拡張
- 1ヶ月後：15事例で安定運用
- 以降：月1回の見直し
