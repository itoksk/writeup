# プロンプト改善チェックリスト

## 📋 週次レビュー用チェックリスト

### 1週目: 基本動作確認
- [ ] 10件以上の承認依頼でテスト実施
- [ ] 判断結果の一貫性を確認
- [ ] 出力フォーマットが正しく守られているか
- [ ] Make連携でエラーが発生していないか
- [ ] 処理時間は許容範囲内か（5秒以内）

### 2週目: エッジケース対応
- [ ] 金額が曖昧な案件での動作確認
- [ ] 複数の条件が重なる案件のテスト
- [ ] 情報不足案件での追加情報要求が適切か
- [ ] 緊急案件での判断速度と品質
- [ ] 例外的な承認依頼への対応

### 3週目: 精度向上
- [ ] 判断理由の説得力を評価
- [ ] 信頼度スコアの妥当性確認
- [ ] 過去の誤判断をプロンプトに反映
- [ ] チームメンバーからのフィードバック収集
- [ ] 成功事例・失敗事例の追加

### 4週目: 最適化と展開
- [ ] 処理速度の最適化（不要な指示の削除）
- [ ] 他部署での試験運用
- [ ] プロンプトのバージョン管理
- [ ] ドキュメントの更新
- [ ] 次月の改善計画策定

## 🔍 品質チェック項目

### 判断の一貫性
- [ ] 同じ条件で同じ判断が出るか
- [ ] 金額基準が正確に適用されているか
- [ ] 優先順位が守られているか

### 出力品質
- [ ] フォーマットが崩れていないか
- [ ] 理由が具体的で分かりやすいか
- [ ] 次のアクションが明確か
- [ ] 文字数制限（500文字）が守られているか

### セキュリティ
- [ ] 個人情報が含まれていないか
- [ ] 機密情報の取り扱いは適切か
- [ ] 金額上限が機能しているか

## 💡 よくある改善ポイント

### プロンプトの改善
1. **曖昧な表現を具体化**
   - ❌ 「適切に判断」
   - ✅ 「ROI 200%以上なら承認」

2. **条件の優先順位を明確化**
   - 法的リスク > 予算超過 > ROI不足

3. **出力形式の厳密化**
   - 改行位置の指定
   - 項目名の統一

### 運用の改善
1. **定期的なプロンプト更新**
   - 月1回の見直し
   - 四半期ごとの大幅改訂

2. **フィードバックループ**
   - 利用者からの意見収集
   - 誤判断の原因分析

3. **バージョン管理**
   - 変更履歴の記録
   - ロールバック可能な体制
