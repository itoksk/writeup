使いやすいアプリは見た目で決まる！
フォームと一覧ビューの整え方

01

講座ゴール

本動画では、5テーブル構成（顧客、社員、サービス、注文、注文詳細）に基づいて、使いやすいフォームと一覧画面を作成します。

学習目標

  • Form ViewとCard Viewの違いを理解する
  • 注文フォームのカスタマイズ方法を学ぶ
  • 注文一覧のカスタマイズ方法を学ぶ
  • 初期値設定とバリデーションルールを実装する
  • 親子関係による関連データ表示を確認する
02

ビュー構成の設計図

5テーブル構成におけるビュー設計の全体像を把握しましょう。

注文テーブル

  • 注文_Form: 新規注文の入力フォーム
  • 注文 (Card View): 注文一覧のカード表示
  • 注文_Detail: 1件の注文の詳細画面

注文詳細テーブル

  • 注文詳細_Form: サービスと数量の入力フォーム
  • 注文詳細 (Table View): 注文詳細の一覧表示

マスタテーブル

  • 顧客_Form: 顧客登録フォーム
  • 社員_Form: 社員登録フォーム
  • サービス_Form: サービス登録フォーム
  • 各テーブルのTable View: 一覧表示

ビュー設計のポイント

1. 操作の目的を明確にする

「閲覧」「登録」「確認」のどれかに絞って、ビューごとに表示項目を調整します。

2. 親子関係を活用する

注文詳細画面では、「Related 注文詳細s」を表示することで、親レコードから子レコードを自動的に表示できます。

3. ステータス別のビューを作成する

同じテーブルに対して、フィルタの異なるビューを複数作成することで、業務の進捗が一目で把握できます。

03

注文フォーム（Form View）のカスタマイズ

ステップ1: 表示項目の選択

自動生成される項目（ID注文、見積番号、請求番号、仮想カラムなど）は非表示にします。

非表示にする項目: - ID注文（自動生成） - 見積番号（自動生成） - 請求番号（自動生成） - 納入単価合計（仮想カラムで自動計算） - 消費税（仮想カラムで自動計算） - 合計金額（仮想カラムで自動計算）

ステップ2: 並び順の調整

業務の入力フローに合わせて、項目の並び順を調整します。

推奨の並び順: 1. ID顧客（参照型：顧客テーブルから選択） 2. ID社員（参照型：社員テーブルから選択） 3. ステータス（見積作成中、見積発行済、受注、納品完了） 4. 発行日 5. 有効期限 6. 納品先 7. 納品先名 8. 受注日 9. 納入期日 10. 納入場所 11. 御支払い条件 12. 備考 13. 振込期限

ステップ3: 初期値の設定

頻繁に入力する項目には、初期値を設定して入力負担を減らします。

発行日の初期値: TODAY() ステータスの初期値: "見積作成中" ID社員の初期値: USEREMAIL()

ステップ4: 必須項目の設定

業務上必須の項目には、Required?をYesに設定します。

必須項目: - ID顧客 - ID社員 - ステータス - 発行日
04

注文一覧（Card View）のカスタマイズ

Primary Header

カードの一番上に表示される最も重要な情報です。

Primary Header: ID顧客 → 顧客名がカードの一番上に表示される
Secondary Header

カードの2行目に表示される補足情報です。

Secondary Header: ステータス → 見積作成中、見積発行済、受注、納品完了などが表示される
Summary Column

カードの3行目以降に表示される詳細情報です。

Summary Column: - 発行日 - 見積番号 - 合計金額 → 注文の概要が一目で把握できる
フィルタ

特定の条件に合致するレコードだけを表示します。

見積作成中一覧のフィルタ: [ステータス] = "見積作成中" 見積発行済一覧のフィルタ: [ステータス] = "見積発行済" 受注一覧のフィルタ: [ステータス] = "受注"
ソート

レコードの表示順序を設定します。

Sort: 発行日 Sort Order: Descending（降順） → 新しい注文が一番上に表示される
05

注文詳細フォームのカスタマイズ

注文詳細テーブルは、注文テーブルの子テーブルです。注文詳細フォームでは、サービスと数量を入力します。

表示項目

  • IDサービス (Ref): サービステーブルから選択
  • 数量: 数値入力

「単価」と「納入金額」は自動計算されるため、入力不要です。

バリデーションルール

  • 数量: Required? = Yes、Min Value = 1
  • IDサービス: Required? = Yes

親子関係による表示

注文詳細画面では、「Related 注文詳細s」という仮想カラムが自動生成され、その注文に紐づく注文詳細が一覧表示されます。

親レコード（注文）から子レコード（注文詳細）を参照: [Related 注文詳細s] 子レコードの合計を計算: SUM([Related 注文詳細s][納入金額])
06

フォーム設計のベストプラクティス

1. 必要最小限のフィールド

自動生成される項目や自動計算される項目は非表示にし、入力が必要な項目だけを表示します。

2. 初期値で入力負担を軽減

発行日=TODAY()、担当者=USEREMAIL()など、頻繁に入力する項目には初期値を設定します。

3. バリデーションで入力ミスを防止

必須項目や最小値・最大値を設定して、不正なデータの入力を防ぎます。

4. ステータス別ビューで進捗を可視化

同じテーブルに対して、フィルタの異なるビューを複数作成します。

5. Card Viewで視認性を向上

Primary Header、Secondary Header、Summary Columnを適切に設定して、一目で内容が把握できるようにします。

6. 親子関係を活用

親レコードから子レコードを自動表示することで、データの整合性を保ちます。

07

初期値設定の例

01

注文テーブルの初期値

発行日: TODAY() → フォームを開いた日付が自動入力される ステータス: "見積作成中" → 新規注文は自動的に「見積作成中」ステータスになる ID社員: USEREMAIL() → ログインしている社員が自動的に選択される 有効期限: TODAY() + 30 → 発行日の30日後が自動入力される
02

注文詳細テーブルの初期値

数量: 1 → デフォルトで1が入力される 単価: [IDサービス].[価格] → 選択したサービスの価格が自動入力される 納入金額: [数量] * [単価] → 数量と単価から自動計算される
08

バリデーションルールの例

注文テーブルのバリデーション

ID顧客: Required? = Yes → 顧客が選択されていない場合、保存できない ID社員: Required? = Yes → 担当者が選択されていない場合、保存できない ステータス: Required? = Yes → ステータスが選択されていない場合、保存できない 発行日: Required? = Yes → 発行日が入力されていない場合、保存できない

注文詳細テーブルのバリデーション

IDサービス: Required? = Yes → サービスが選択されていない場合、保存できない 数量: Required? = Yes, Min Value = 1 → 数量が1以上でない場合、保存できない

カスタムバリデーション式の例

より複雑なバリデーションが必要な場合は、Valid_If式を使います。

有効期限が発行日より後であることを確認: [有効期限] > [発行日] エラーメッセージ: "有効期限は発行日より後の日付を入力してください"
09

よくある失敗例と対策

失敗例1

すべてのフィールドを表示してしまう

対策: 自動生成される項目や自動計算される項目は非表示にする。入力が必要な項目だけを表示する。

失敗例2

初期値を設定していない

対策: 発行日=TODAY()、担当者=USEREMAIL()など、頻繁に入力する項目には初期値を設定する。

失敗例3

バリデーションルールが不十分

対策: 必須項目や最小値・最大値を設定して、不正なデータの入力を防ぐ。

失敗例4

Card ViewのPrimary Headerが適切でない

対策: 最も重要な情報（顧客名など）をPrimary Headerに設定する。

失敗例5

ステータス別のビューを作成していない

対策: 同じテーブルに対して、フィルタの異なるビューを複数作成する。

失敗例6

親子関係を活用していない

対策: 注文詳細画面で「Related 注文詳細s」を表示して、親レコードから子レコードを自動表示する。
