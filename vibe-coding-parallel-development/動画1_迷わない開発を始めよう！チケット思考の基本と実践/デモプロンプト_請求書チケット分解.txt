# 🎯 チケット思考デモプロンプト集：請求書管理システムの開発タスク分解

## ❌ ダメな例（曖昧すぎる指示）

```
請求書管理システム作って
```

### AIの反応：
「どのような機能が必要ですか？技術スタックは？規模感は？」
→ 質問攻めで進まない、何から始めればいいか分からない

---

## ⭕ 良い例（チケット形式で分解）

```
請求書管理システムを作りたいです。
必要な機能をチケット形式で分解してください。

各チケットには以下を含めてください：
- 目的（1行）
- 完了条件（箇条書き3つ）
- 見積時間（30分〜3時間）
- 優先度（高/中/低）

まず基本機能5つをお願いします。
```

### AIの出力例：
```
チケット #001：データ構造の設計【優先度：高】
目的：請求書データの保存先を準備する
完了条件：
- スプレッドシートに必要な列が作成されている
- データ型が定義されている
- テストデータが1件入っている
見積時間：30分

チケット #002：入力フォームの作成【優先度：高】
目的：請求書データを入力する画面を作る
完了条件：
- HTMLフォームが表示される
- 必須項目（顧客名、金額、日付）が入力できる
- 送信ボタンが動作する
見積時間：2時間

チケット #003：データ保存機能【優先度：高】
目的：入力データをスプレッドシートに保存する
完了条件：
- フォームデータを受け取れる
- スプレッドシートに追記される
- 保存成功メッセージが表示される
見積時間：1.5時間

チケット #004：一覧表示機能【優先度：中】
目的：保存した請求書を確認できるようにする
完了条件：
- 全請求書のリストが表示される
- 日付順にソートされている
- 件数が表示される
見積時間：1時間

チケット #005：請求書番号の自動採番【優先度：低】
目的：請求書に一意の番号を自動付与する
完了条件：
- INV-YYYY-XXXX形式で採番される
- 重複しない番号が生成される
- 既存データと整合性が取れる
見積時間：45分
```

## 📊 チケット分析プロンプト

```
開発チケットを分析して改善点を教えてください。

チケット履歴：
- 見積30分 → 実績90分：「認証機能実装」
- 見積60分 → 実績45分：「データ保存」
- 見積120分 → 実績180分：「UI全体作成」

何が問題で、どう改善すればいいですか？
```

### AIの分析例：
```
問題点：
1. 複雑な機能の見積が甘い（認証、UI）
2. チケットが大きすぎる可能性

改善策：
1. 複雑な機能は1.5〜2倍の見積に
2. 2時間超えるチケットは分割
3. 「調査」と「実装」を別チケットに
```

## ✅ チケット完了条件の明確化

```
「ユーザー認証機能」のチケットに対して、
明確な完了条件を5つ設定してください。

テスト可能で、Yes/Noで判断できる条件で。
```

### AIの出力：
```
□ メールアドレスとパスワードでログインできる
□ ログイン失敗時にエラーメッセージが表示される
□ ログイン成功後、セッションが保持される
□ ログアウトボタンでセッションが破棄される
□ 未ログイン時は自動的にログイン画面にリダイレクトされる
```

## 🎮 チケット思考のベストプラクティス

1. **1チケット = 1成果物** - 明確な完了状態
2. **半日以内で完了** - 大きすぎると管理困難
3. **依存関係を明示** - 実装順序が明確
4. **完了条件は測定可能** - 曖昧さを排除
5. **定期的に振り返り** - 見積精度の改善

**Remember: 小さな完了の積み重ねが、大きな成果につながる！**
